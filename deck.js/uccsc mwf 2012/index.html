An Update On MWF Collaborative Efforts

UCSF Mobile Initial Launch

In 2010, we launched UCSF Mobile. It was kind of a lark. No one was doing a mobile site for UCSF so we just went and did it. It was far more successful than we expected and we decided to get ambitious. We wanted perhaps to bite off more than we could chew. The MWF project had some nice features we didn't and other UCs were adopting it, so it seemed like a no-brainer for us to join in.

UCSF Mobile 2.0

Our site looked much better and we got to contribute lots of stuff that helped others: tests, features such as a customizable home screen, bug fixes, and so on. We were very active and it was a lot of fun. Our particular need, though, was for performance, and for performance in ways that perhaps other schools didn't need it. You see, our web site was also our interface for our native iPhone, iPad, and Android apps. You can't have a "click the button, and wait for 3 and a half seconds before you see content" experience.

Focus on performance

  • MWF appcache handler
  • Lightning Touch (http://trott.github.com/LightningTouch)
So, we really targeted performance. ... We contributed code for an appcache handler to MWF so that sites could run offline and load faster when they were online. ... We released a library called Lightning Touch that I encourage you to check out. It makes the touch interface faster. These helped a lot ... but we wanted to take it further. We want the fastest site possible. By the way, these changes not only make the native app feel like, well, a native app, but they also greatly improve the user experience in the browser. I think this is awesome, because why should web users be treated like second class citizens?

MWF's Device Telemetry Stack

  • Makes telemetry information available on the server.
  • Support for distributed sites (content is on one server and MWF is on another server) means redirects.
We took some measurements and determined that performance took its biggest hit on initial page load from the Device Telemetry Stack or DTS. ... DTS makes information like screen size available on the server. ... The DTS implementation uses a pair of redirects on initial load. ... Redirects hurt your page load speed. On mobile, the delay can be measured in seconds.

DTS is deeply embedded in MWF

Can we really remove it?

Because DTS is so deeply embedded in MWF, the question that came up was: ... Are we biting off more than we can chew? Can we really remove it? ... The answer turned out to be: Yes, we can! There are a lot of features in DTS, but the truth is that we just weren't using most of them. And the ones that we were using, we could accomplish the same thing in other ways.

Site speed FTW!

The site is now plenty fast. ... And a nice side benefit is that this resulted in a much more svelte code base. Now, basically, once we had removed DTS, we in effect weren't really running MWF anymore. In an unhealthy collaborative environment, this would be the end of the collaboration. But we stayed involved, kept talking to everyone, and our interests along with UCSD's different interests eventually gelled with UCLA's plans and poof!

MWF 2.0

MWF 2.0 ... will be device telemetry stack-free. It will also have responsive web design features.

Collaboration is overrated

Ha, ha! Just kidding!

The moral of the story, of course, is that collaboration is totally overrated. ... No, just kidding.

Collaboration

We would have never gotten as far as we did without MWF as a jumping-off point. We would not have had as many things to experiment with and would not have had as many people to bounce ideas off. Both the spirit of collaboration and, yes, healthy competition enabled us to bite off more than we could chew, and then proceed to chew it.
