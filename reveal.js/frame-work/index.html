Why are we even talking about this?

I'm going to focus on frameworks and technology because talking about is easy and fun. The hard parts of an edu mobile site tend to be insuring high-quality content, how to do user testing, how to measure engagement, getting high-quality content, how to keep the content and experience fresh and engaging, how to keep the content from appearing dated six months from now, and content. Whether to use Kurogo or jQuery Mobile is, comparatively speaking, unimportant. It's an implementation detail.

With that said, let's get down to the unimportant stuff.

UCSF's initial mobile web site launched in spring 2010. It was put together using CodeIgniter, an MVC web framework in PHP. After reviewing UC Mobile Framework, Kurogo, and jQuery Mobile, we moved our mobile web to the UC Mobile Web Framework and contributed code for features such as their AppCache handler for offline use and automated acceptance tests. In an effort to make our mobile web experience as performant as possible, we also created small but useful tools like the open source Lightning Touch library (go to trott.github.com). Over time, our site has diverged sufficiently from MWF that we kinda sorta aren't really running MWF. In particular, we could not countenance the performance hit of what the MWF developers refer to as the Device Telemetry Stack or DTS. Removing it from our instance resulted in a cascade effect of MWF code being disabled or removed from our site.

UCSF Mobile native app (iOS, Android)

  • Blackboard Mobile
  • Blackboard Mobile MWF
We used Blackboard to create our initial UCSF Mobile app. Subsequently, we explored making native apps available via PhoneGap and Appcelerator Titanium, but at the time, the technology was sufficiently immature that it made sense to use the native wrappers in UC Mobile Framework instead. This is what we are currently using.

General Directions

Server Technology
Client Technology
Broadly speaking, we're moving away from a reliance on server-side logic (which is pretty central to UC Mobile Framework and Kurogo implementations) and towards putting the logic on the client end.
Server Libraries (e.g., SimplePie)
Templates (e.g., Mustache)
For example, our news readers rely on client-side JS templating and read items directly from RSS feeds rather than relying on aserver-side library to parse and render from RSS.
Server-side caching
AppCache & LocalStorage
The client caches the RSS content in localStorage, enabling offline access. This is useful for events calendars that are served via RSS, for example.
