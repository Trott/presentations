Why are we even talking about this?

I'm going to focus on frameworks and technology because talking about is easy and fun. The hard parts of an edu mobile site tend to be insuring high-quality content, how to do user testing, how to measure engagement, getting high-quality content, how to keep the content and experience fresh and engaging, how to keep the content from appearing dated six months from now, and content. Whether to use Kurogo or jQuery Mobile is, comparatively speaking, unimportant. It's an implementation detail.

With that said, let's get down to the unimportant stuff.

UCSF's initial mobile web site launched in spring 2010. It was put together using CodeIgniter, an MVC web framework in PHP. After reviewing UC Mobile Framework, Kurogo, and jQuery Mobile, we moved our mobile web to the UC Mobile Web Framework and contributed code for features such as their AppCache handler for offline use and automated acceptance tests. In an effort to make our mobile web experience as performant as possible, we also created small but useful tools like the open source Lightning Touch library (go to trott.github.com).

UCSF Mobile native app (iOS, Android)

  • Blackboard Mobile
  • Blackboard Mobile MWF
We used Blackboard to create our initial UCSF Mobile app. Subsequently, we explored making native apps available via PhoneGap and Appcelerator Titanium, but at the time, the technology was sufficiently immature that it made sense to use the native wrappers in UC Mobile Framework instead. This is what we are currently using.
Broadly speaking, we're moving away from a reliance on server-side logic (which is pretty central to UC Mobile Framework and Kurogo implementations) and towards putting the logic on the client end. For example, our news readers rely on client-side JS templating to read items from RSS feeds rather than a server-side library to parse and serve the RSS. The client caches the RSS content in localStorage, enabling offline access. This is useful for events calendars that are served via RSS, for example. Ultimately, we may may choose to go with one of the many MV* JS frameworks if performance doesn't suffer as a result. On the native wrapper side, we'll probably give PhoneGap another look.
Our forward movement in the mobile web space has slowed somewhat of late as we do not have resources dedicated to it, but I'm optimistic that we'll see some change there.
